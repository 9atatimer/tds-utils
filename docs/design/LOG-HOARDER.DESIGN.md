# LOG-HOARDER.DESIGN.md

> This document supersedes `SEMANTIC-SEARCH.DESIGN.md`, which was written when
> txtai was treated as MVP scaffolding and the hexagonal port was justified by
> a planned swap to direct sqlite-vec. Both of those positions have been
> reversed. The earlier doc is retained for historical context only.

## 1. Overview

`log-hoarder` captures every tmux pane's terminal output to disk and makes
that history searchable by meaning, not just by keyword. The capture side
(tmux hooks, `tmux_logging.sh`, `tmux_shepherd.sh`, archive sweeping) is
established and is **not** changed by this design.

What this document specifies is the search side: how those captured logs
become a queryable semantic index, and how that index is exposed to the user.

The operational model is a **long-lived local daemon, `log-hoarderd`**,
managed by launchd. The daemon owns the search index, holds a persistent HTTP
client to Ollama (the embedding and LLM provider), and exposes a localhost
HTTP API for indexing and search. CLI tools (`log_search`, `log_ask`) are
thin clients that POST to the daemon. The tmux capture machinery notifies the
daemon when new pane directories are ready to index. There is no per-process
model loading, no cron-driven indexing, and no in-process embedding.

The search backend is **txtai**, configured to use `sqlite-vec` for vector
storage and Ollama for embedding generation. This is a deliberate, long-term
commitment: see §4.

## 2. Terminology

- **Pane Directory** — a tmux pane's archived log directory, located at
  `$TDS_LOG_DIR/archived/SESSION/WINDOW/PANE/`. The unit of capture and the
  unit of ingestion.
- **Log File** — a single `.log` file inside a pane directory. A pane
  directory may contain more than one if the pane was reused across tmux
  sessions; each is a distinct byte-offset space.
- **Chunk** — a token-window slice of a log file, sized for the embedding
  model. The unit of embedding and the unit of retrieval.
- **Span** — a logically coherent stretch of activity within a session.
  Deferred. Not present in the index. See §7.
- **Slug** — a short human-readable label for a pane directory, generated by
  a local LLM (`log_brander`) from sampled log content. Used for display and
  optionally for embedding (§11).
- **Sidecar** — a JSONL file emitted by tmux hooks alongside each pane log,
  capturing per-command metadata (`cwd`, `venv`, `exit_code`, timestamp) that
  the shell prompt does not otherwise preserve.

## 3. Core Principles

These are the load-bearing constraints that the rest of the design follows
from. Each is stated as a rule and each rules out a class of code.

1. **Source of truth lives on disk.** The archived `.log` files are the
   single source of truth. The search index is an optimization over them and
   can be rebuilt from them at any time.

2. **Full coverage.** Every byte of every archived log file is covered by
   embeddings. There is no sampling, no head/tail truncation, no
   "representative" content. The current `get_log_sample` scaffolding is
   removed.

3. **The index is not the display source of truth.** Chunk text is cached
   in the txtai content store because hybrid retrieval (BM25 alongside dense
   vectors) needs it at query time — sparse scoring cannot work on text the
   index does not hold. But that cached text is *only* used for scoring,
   never for display. Previews shown to the user are read from the source
   `.log` file at byte offset, never from the index. This preserves the
   intent (no risk of the index drifting from on-disk truth, no risk of
   stale text surviving a log rewrite) without fighting the framework.

4. **Schema bumps trigger full reindex.** Because SOT is on disk and reindex
   is cheap, there are no in-place schema migrations. The daemon records the
   schema version it was built under; mismatch on startup triggers a
   `DROP TABLE; reindex everything` flow. This is a feature, not a
   limitation — it eliminates an entire category of versioning code. (This
   assumption holds while archived log volume stays in the GB range. If
   retention ever grows to TB-scale, schema bumps would need to become
   incremental — but that is a problem for whoever has terabytes of terminal
   history, not for us today.)

5. **Per-pane ingestion state lives next to the pane.** A pane directory is
   "ingested" iff it contains a `.indexed` marker file written atomically
   after all its chunks are committed to the index. The daemon's "what's
   pending?" query is a filesystem walk, not a database query. If the index
   is destroyed, removing all `.indexed` markers is a one-liner and the
   daemon repopulates. Crash recovery is automatic: a pane that was
   mid-ingest at crash time has no marker, so it looks pending and is redone
   on restart. Idempotency by construction.

6. **Ollama is forever, and Ollama failures fail loud.** Both local and
   remote Ollama are treated as permanent infrastructure. When either is
   unreachable, the daemon does not queue, retry with backoff, or fall back
   silently. It logs loudly, marks itself unhealthy on the health endpoint,
   and stops processing until the human fixes it. The filesystem is the
   queue: missing `.indexed` markers represent the work that needs to be
   redone when the daemon comes back up.

## 4. Architecture

Three layers, in order from innermost to outermost.

### 4.1 Domain

Pure Python, no I/O, no framework dependencies. Lives in `search/domain/`.

- **Models** — `LogEntry`, `Chunk`. The existing `Span` and
  `Chunk.from_span()` are retained on disk but unused by the indexing
  pipeline; they are inert until §7's deferral is reconsidered.
- **Ports** — `SearchIndexPort` (store and query embeddings) and
  `EmbeddingPort` (generate an embedding vector from text). Both are
  abstract.

The hexagonal port is **retained as insulation, not as swap-readiness.** It
isolates the rest of the daemon from txtai's API surface so version upgrades,
configuration changes, and txtai's own evolution don't ripple. It does not
exist because we plan to swap txtai for something else. We don't.

### 4.2 Adapters

One adapter per port. Lives in `search/adapters/`.

- **`TxtaiAdapter`** implements `SearchIndexPort`. It wraps a txtai
  `Embeddings` instance configured with `backend="sqlite"` (sqlite-vec under
  the hood), `content=True` (SQLite content store for metadata columns), and
  `hybrid=True` (BM25 sparse index alongside dense vectors — terminal logs
  benefit from exact matches on filenames, flags, and error codes).
- **`OllamaEmbedder`** implements `EmbeddingPort`. It POSTs to Ollama's
  `/api/embed` endpoint, local first with remote-Ollama as failover. The
  failover policy is part of this adapter, not the daemon.

The `TxtaiAdapter` is configured to delegate vector generation to
`OllamaEmbedder` rather than loading sentence-transformers in-process. This
is the single most important configuration choice in the design: it removes
the only reason the search process would ever need to import torch.

### 4.3 Daemon

`log-hoarderd` is a long-lived Python process, managed by launchd, that
holds the warm txtai instance and exposes an HTTP API on localhost. Lives in
`search/daemon/`.

Responsibilities:
- Own the `TxtaiAdapter` instance and its in-memory state.
- Hold a persistent HTTP client to Ollama (keepalive, connection reuse).
- Serve indexing requests (POST a pane directory path → ingest it).
- Serve search requests (POST a query → return ranked chunk hits).
- Serve RAG requests (POST a question → return generated answer with
  citations).
- Expose a health endpoint reflecting Ollama reachability and recent error
  state.
- Maintain in-memory metrics counters exposed via `/health` — embed
  calls/minute, index writes/minute, search calls/minute, error counts by
  category. This is the day-one instrumentation that §11 question 1
  depends on for deciding whether live-pane indexing needs debouncing.
- On startup: check schema version, trigger full reindex if mismatched.

The daemon is the *only* writer to the index. CLI clients are read-only or
indirect-write (POST to `/index-pane`). This eliminates lock contention and
makes ingestion ordering trivially serial.

### 4.4 Why txtai

Worth naming honestly, because the previous design treated txtai as a
disposable scaffold. The whole-hog commitment is justified by features that
are configuration in txtai and meaningfully more work without it:

- **`backend="sqlite"`** gets us sqlite-vec for free. The "txtai vs
  sqlite-vec" question dissolves: txtai-on-sqlite-vec is both decisions taken
  at once.
- **SQL with `similar()`** lets metadata filters (`cwd LIKE ...`,
  `exit_code = 0`, `timestamp > ...`) compose with semantic search in a
  single query. This is the natural query interface for the sidecar metadata
  in §6.
- **Hybrid sparse+dense** with one config flag.
- **`RAG(embeddings, llm)`** is one constructor away once the daemon is up.
  This is what enables `log_ask`, which is qualitatively different from
  search.
- **Workflow framework** gives the indexing pipeline a structured home (§5)
  and pre-built stages for things we'll want eventually (summarization for
  branding, document conversion if we ever index files alongside logs).
- **FastAPI server bundled** — txtai ships a YAML-configured API server. The
  daemon may use it directly or wrap it; either way the HTTP plumbing is not
  hand-rolled.

The cost of this commitment: txtai's import time is non-trivial (torch is in
its transitive closure even when not used for embedding), which is why the
daemon model is mandatory rather than optional. This trade is acceptable
because the daemon is wanted on its own merits — see §9.

## 5. Indexing Pipeline

Indexing is structured as a txtai workflow: a declared sequence of stages
that data flows through. The daemon registers the workflow at startup and
invokes it per pane directory, either in response to a notification from
`tmux_shepherd.sh` or during a startup reindex sweep.

### 5.1 Stages

1. **Discovery.** Walk `$TDS_LOG_DIR/archived/`, yield pane directories that
   lack a `.indexed` marker. Used during startup reindex and as a periodic
   sweep for stragglers (panes that were archived while the daemon was
   down).

2. **File reading.** For each pane directory, enumerate its `.log` files.
   Reading goes through a single file-access abstraction so that compressed
   logs (§11) can be supported in one place later. Byte offsets recorded by
   downstream stages always reference *uncompressed* content.

3. **Cleaning.** Strip residual ANSI escape sequences and control characters
   (post-`ansifilter`, some still leak through), collapse runs of whitespace,
   discard tab-completion artifacts. Cleaning preserves byte offsets into
   the original file — the cleaned text is for embedding only, not for
   storage or display.

4. **Chunking.** txtai's token-window chunker, sized to `nomic-embed-text`'s
   context window with overlap. Chunks do not respect any logical
   boundaries — there are no spans (§7), and the boundary-fidelity question
   is deferred until empirical retrieval quality demands otherwise.

5. **Embedding.** Each chunk is sent to Ollama via `OllamaEmbedder`. The
   adapter's local-first / remote-failover policy applies. If both are
   unreachable, the entire pipeline aborts loudly per §3 rule 6.

6. **Indexing.** Embedded chunks are upserted into the txtai index with
   their full metadata payload (§6). Upsert (not insert) so that re-running
   the pipeline against an already-partially-ingested pane is safe.

7. **Marker write.** After all chunks for a pane directory are committed,
   write `.indexed` atomically (write to `.indexed.tmp`, rename). The
   presence of this file is the sole authoritative signal that the pane is
   fully ingested.

### 5.2 Pipeline shape

The pipeline is per-pane-directory, not per-chunk and not per-file. A pane
directory is the unit of "did this finish successfully?" because that's the
unit the `.indexed` marker lives at. Within a pane directory, processing of
multiple log files and their chunks is an internal detail of the pipeline.

If the pipeline aborts partway through a pane directory (Ollama dies,
process crash, etc.), no marker is written, and the entire pane is redone on
the next pass. Partial progress is discarded. This is acceptable because
panes are small (KB to single-digit MB) and reprocessing is cheap.

## 6. Schema

One row per chunk. Stored in the txtai SQLite content store; the embedding
itself lives in the sqlite-vec virtual table that txtai manages alongside it.

| field | type | source | purpose |
|---|---|---|---|
| `id` | string PK | `{session_path}:{log_file}:{chunk_index}` | stable identity for upsert idempotency |
| *(embedding)* | vector | computed from chunk text | the search payload, managed by sqlite-vec |
| `session_path` | string | filesystem | pane directory path; groups chunks by pane and locates source files |
| `log_file` | string | filesystem | which `.log` file within the pane directory; required because byte offsets are per-file |
| `chunk_index` | int | chunker output | ordering within the log file |
| `byte_start` | int | chunker output | offset into uncompressed log; preview targeting |
| `byte_end` | int | chunker output | ditto |
| `cwd` | string nullable | sidecar | filter dimension ("only chunks from this repo") |
| `venv` | string nullable | sidecar | filter dimension |
| `exit_code` | int nullable | sidecar | filter dimension ("only successful commands") |
| `host` | string | sidecar | the machine's self-reported hostname at command time |
| `timestamp` | int (unix seconds) | sidecar or log line | range filter, ordering |
| `slug` | string nullable | `log_brander` | display-time enrichment, optionally also embedded |
| `metadata_confident` | bool | chunker output | `true` when the chunk's metadata came from an exact `cmd_seq` marker match; `false` when correlation had to fall back to a prior marker. See §11 question 4 |
| `schema_version` | int | constant | reindex trigger on mismatch |

### 6.1 Notes on the choices

- **`text` is stored in the index for hybrid retrieval.** txtai's `hybrid=True`
  mode runs BM25 over the content store's `text` column alongside the dense
  vector search. BM25 cannot score against text the index does not hold, so
  the cached text is required for hybrid to function. Per §3 rule 3, this
  cached text is used *only* for sparse scoring — never read back for
  display. Previews come from the source `.log` file at byte offset. The
  index ends up roughly 2-3× larger than an embeddings-only version would
  be; for terminal logs this is tens of MB per heavy-use year, which is
  acceptable in exchange for hybrid retrieval working at all.

- **Sidecar metadata is denormalized into every chunk row.** A chunk
  inherits its `cwd`/`venv`/`exit_code`/`host`/`timestamp` from the sidecar
  event closest in time to its byte range. The bloat is negligible compared
  to the embedding vectors, and every filter becomes a single-table `WHERE`
  with no joins.

- **`slug` is duplicated across every chunk in a pane directory.** Same
  reasoning. Branding happens once per pane and updating ~50 rows is fine.

- **No `session` table, no `pane` table.** Anything you'd want to know about
  a pane as a whole (duration, total chunks, dominant cwd) is either
  computable on demand from the chunk rows or stashed in a JSON file inside
  the pane directory itself, next to `.indexed` and `slug.txt`.

- **The composite primary key includes `log_file`** because a single pane
  directory can hold multiple log files with independent byte-offset spaces.
  Upserts must distinguish between them.

### 6.2 Query shape

The natural query becomes:

```sql
SELECT session_path, log_file, byte_start, byte_end, score, slug, timestamp
FROM txtai
WHERE similar(:query)
  AND cwd LIKE :cwd_prefix || '%'
  AND exit_code = 0
  AND timestamp > :since
ORDER BY score DESC
LIMIT 50
```

Grouping by pane directory (so the user sees one result per pane, not five
adjacent chunks from the same pane) happens in Python after the SQL
returns: fetch top-N chunks, collapse to top-M panes by best-chunk score.
This is the same post-processing that span-based grouping would do, just
without spans.

## 7. Spans: Deferred

The `Span` and `Chunk.from_span()` machinery in `search/domain/models.py`
is preserved on disk but is not wired into the indexing pipeline. The
indexing path goes straight from cleaned text to fixed-size chunks, with no
span detection.

This is a deliberate deferral, not an oversight. The reasoning:

- **Spans were a means, not an end.** The end is "I type a query and land
  in the right pane at roughly the right place." If naive chunking + hybrid
  retrieval already achieves that, spans are speculative engineering against
  a problem we don't have.
- **The question is empirical, not architectural.** It cannot be answered
  from a whiteboard. It is answered by indexing real logs with the simple
  approach, running real queries, and looking at what comes back.
- **Deferral costs nothing.** The dataclasses are written and tested; if
  they're needed, they're waiting. The hexagonal port absorbs the schema
  change cleanly when and if that day comes.

Three scenarios would bring spans back:

1. **Display.** Preview snippets need more context than a single chunk. A
   span — bounded by prompt lines on either side of the matching chunk — is
   the natural display unit. This can be computed at query time from the
   source file without storing spans in the index.
2. **Deduping.** If grouping by pane directory is too coarse and grouping
   by chunk is too fine, spans become a useful intermediate. Also doable
   query-time.
3. **RAG context packing.** If `log_ask` needs to feed the LLM "the whole
   logical unit" rather than individual chunks, spans give it the right
   boundary. This is the only scenario where spans might want to be
   precomputed, and only if query-time computation proves too slow.

In all three cases, spans can be re-introduced without a schema change.
The chunk row already carries `byte_start`, `byte_end`, `session_path`, and
`log_file` — everything needed to reconstruct spans on demand.

## 8. Query Path

### 8.1 Search

A search request flows: `log_search "query"` → HTTP POST to daemon → daemon
embeds the query via `OllamaEmbedder` → daemon executes the SQL above
against the txtai index → daemon groups chunk hits by pane directory →
daemon reads source files at the matching byte ranges to extract preview
snippets → daemon returns a JSON response → `log_search` formats results for
fzf.

The daemon — not the shell client — extracts preview snippets, because the
daemon already has file-reading capability and knows the byte offsets. The
shell client only renders.

### 8.2 Output contract

The daemon returns one JSON object per result, with at minimum:

- `pane_dir` — absolute path to the pane directory
- `log_file` — absolute path to the specific `.log` file
- `byte_start`, `byte_end` — for `$PAGER` positioning and preview targeting
- `score` — for ordering and debugging
- `timestamp` — for display
- `slug` — for display, may be null
- `cwd` — for display, may be null
- `preview` — pre-extracted snippet of the matching region, plain text

The shell layer (`log_search` + the ZLE widget) renders this into fzf and
handles selection. The wire format is HTTP+JSON, not a custom socket
protocol, because the daemon already speaks HTTP for indexing and there's
no reason to invent a second transport.

### 8.3 CLI behavior when the daemon is unhealthy

The daemon's fail-loud stance (§9.4) is only useful if the CLI tools
surface that loudness to the human. When `log_search` or `log_ask`
receives a 503 from the daemon, it must:

1. Print the daemon's diagnosis to stderr. The 503 response body carries
   the relevant fields from `/health` — Ollama local/remote reachability,
   pending pane count, last successful index timestamp, recent error
   count.
2. Exit non-zero (exit code 2 for daemon-unhealthy, distinct from exit 1
   for "no results").
3. **Not** fall back to anything. No degraded mode, no in-process
   embedding, no cached results from a previous successful query. The
   user sees the failure and fixes the underlying problem.

What the user sees:

```
log_search: daemon unhealthy
  ollama local:  unreachable (last try: 14s ago)
  ollama remote: unreachable (last try: 14s ago)
  pending panes: 17
  last successful index: 4h12m ago
exit 2
```

If the daemon itself is not running (connection refused, not 503), the
CLI prints a similarly clear message pointing at launchd and the daemon
log path, and exits non-zero.

This is the user-facing manifestation of "fail loud." Silent empty
results from an unhealthy daemon would be the worst possible outcome —
the user would conclude "my history doesn't contain that" when the real
answer is "the index couldn't tell you."

### 8.4 RAG

`log_ask "question"` is a peer to `log_search`, not an afterthought. It
POSTs to a separate daemon endpoint, which instantiates (or reuses) a
`RAG(embeddings, llm)` object pointed at Ollama for both retrieval and
generation, and returns `{answer, citations}` where each citation is a
`(pane_dir, log_file, byte_start, byte_end)` tuple the shell client can
expand into a `$PAGER` invocation.

This endpoint exists from day one of the daemon, because once the daemon
is running and txtai is loaded, adding RAG is configuration, not code. The
marginal cost is near zero and the marginal value is high — it's the
qualitative leap that justifies having built a daemon at all.

## 9. Daemon Operations

### 9.1 Lifecycle

- **Started by:** launchd, via a plist in `~/Library/LaunchAgents/`.
- **Logs to:** `~/Library/Logs/log-hoarder/{daemon.log, daemon.err}`.
- **Binds to:** `127.0.0.1:<port>`, localhost only. No auth — the bind
  address is the access control.
- **State on disk:** `$TDS_LOG_DIR/index/` (the txtai index directory) plus
  `.indexed` markers scattered through `$TDS_LOG_DIR/archived/`.
- **Restart policy:** launchd `KeepAlive`, unconditional.

### 9.2 Startup sequence

1. Read schema version from the existing index, if any.
2. If absent or mismatched: drop the index, walk `archived/` removing all
   `.indexed` markers, log the reindex decision loudly.
3. Load (or create) the txtai index.
4. Verify Ollama reachability (local first, then remote). If neither is
   reachable, log loudly and start the HTTP server in unhealthy state — do
   not block startup, because the human needs the health endpoint to
   diagnose what's wrong.
5. Begin serving HTTP. Kick off a background sweep for pane directories
   without `.indexed` markers.

### 9.3 Health

A `GET /health` endpoint returns:

- Daemon uptime
- Schema version
- Ollama local reachable (bool, last-checked timestamp)
- Ollama remote reachable (bool, last-checked timestamp)
- Pending pane directories (count)
- Recent error count
- Last successful index timestamp

This is the human's diagnostic surface when something is wrong. It is also
the surface a future status-line widget would query.

### 9.4 The "fail loud" stance, made concrete

When Ollama becomes unreachable mid-pipeline:

1. The current pane being indexed aborts. No `.indexed` marker is written.
2. The error is logged at ERROR level with full context.
3. The health endpoint flips to unhealthy and stays there.
4. The daemon stops accepting new index requests (returns 503).
5. Search and RAG endpoints continue serving from the existing index — they
   don't need Ollama for retrieval, only for query embedding, so query
   embedding will also start failing with 503.
6. When Ollama comes back, the human pokes the daemon (or the daemon
   notices on its periodic health check), and the background sweep picks
   up everything still missing a marker.

No exponential backoff. No dead-letter queue. No circuit breaker. The
filesystem is the queue.

## 10. Integration with tmux Capture

The capture side (`tmux_logging.sh`, `tmux_shepherd.sh`, the tmux hooks) is
mostly unchanged. Two additions and one new piece of plumbing:

- **`tmux_shepherd.sh`** currently archives pane directories on
  `pane-exited` and runs branding via cron. The shepherd gains one
  responsibility: after archiving a pane directory, fire-and-forget POST to
  `http://127.0.0.1:<port>/index-pane` with the pane directory path. If the
  daemon is down, the POST fails and is logged; the pane sits in
  `archived/` without an `.indexed` marker until the next daemon startup
  sweep picks it up. No retry logic in the shepherd.
- **The branding cron job goes away.** Branding moves into the daemon as a
  workflow stage that runs on the same warm Ollama connection the indexer
  is already using. `log_brander` as a standalone shell script is retained
  for manual one-off use but is no longer wired into cron.
- **The shell prompt emits a correlation marker.** A `precmd` hook in
  `dot.zshrc` writes a single line of the form `‹cmd:N›` to the pane's
  output stream just before each prompt is drawn, where `N` is a
  per-pane monotonic counter. This marker is what the chunker stage uses
  to anchor sidecar metadata (cwd, venv, exit_code, host) to byte
  positions in the log. See §11 question 4 for the full correlation
  strategy. The sigil characters are chosen to be unlikely to appear in
  real terminal output and are filtered out of any text shown to the
  user. This is the only capture-side change required by this design.

What does *not* change:

- Archive sweeping. Still in `tmux_shepherd.sh`, still shell, still runs on
  `pane-exited` and orphan sweep. The daemon is not in this path because
  archiving is filesystem work that should not depend on Python being up.
- The directory layout under `$TDS_LOG_DIR`.
- The `TDS_LOG_DIR` environment variable contract.

## 11. Open Questions

These are pinned for resolution at implementation time, not now.

1. **Live-pane indexing debouncing.** Once the daemon is running, indexing
   active panes (not just archived ones) becomes feasible — the daemon can
   tail open log files and index incrementally. This could thrash Ollama if
   done naively. Resolution: instrument from day one (embed-calls-per-minute
   counter on `/health`), build something only if real workload shows a
   problem. Not over-engineering until the data demands it. **When this is
   built, it must go through the same daemon-internal writer path as
   archived-pane indexing.** The "daemon is the only writer" invariant from
   §4.3 is preserved regardless of whether the trigger is a shepherd POST
   or a tail-driven internal event.

2. **Compressed log storage.** Carried forward from the previous design.
   Logs can remain uncompressed for now, but the file-access abstraction in
   §5 stage 2 is the chokepoint where compression will eventually be added.
   Unresolved sub-questions:
   - Format: gzip (ubiquity) vs zstd (ratio + seekability).
   - Granularity: per-file or per-pane.
   - Random access: zstd seekable format, or always-decompress-fully?

3. **Slug embedding strategy.** Should the slug be embedded as part of the
   first chunk of a pane (so semantic queries hit it), or stored display-only?
   Current lean: embed it, by concatenating slug into the first chunk's text
   before embedding. Decision deferred until retrieval quality is measured
   with and without.

4. **Sidecar JSONL correlation strategy.** The chunker stage needs to
   consume both the log file *and* the sidecar JSONL stream, correlating
   sidecar events to byte positions in the log so each chunk inherits the
   right `cwd`/`venv`/`exit_code`/`host`. The strategy:

   - **Sidecar emission.** The tmux `after-send-keys` hook writes one
     JSONL record per command-submitting keystroke, with fields:
     `{ts, pane, cwd, venv, host, cmd_seq}`. `cmd_seq` is a monotonic
     per-pane counter — this is the load-bearing field, because it gives
     us a correlation key that does not depend on timestamp matching.
   - **Log emission.** The shell `precmd` hook emits a marker line of the
     form `‹cmd:N›` into the pane output just before each prompt, where
     `N` matches the sidecar's `cmd_seq`. The sigil is chosen to survive
     `ansifilter` and to be implausible in real output. See §10.
   - **Correlation at index time.** The chunker scans for `‹cmd:N›`
     markers in the cleaned log text and uses them as anchors. Each chunk
     inherits the metadata of the highest `cmd_seq` marker at or before
     its `byte_start`. The match is integer equality on `cmd_seq` — exact,
     not fuzzy. Clock skew is not in the loop because clocks are not in
     the loop.
   - **Missing-marker handling.** If a marker is dropped (the sidecar hook
     fired but the log line was eaten by some terminal weirdness, or vice
     versa), the chunk inherits from the previous successful marker and
     gets `metadata_confident` set to `false`. The searcher can consult
     this when the user adds a metadata filter (`--in <repo>`) to either
     drop or de-rank uncertain rows. The schema needs one extra boolean
     column — added now to avoid a schema bump later when this case
     actually shows up.

   The remaining unknown is purely the choice of sigil characters for the
   marker line, which is an implementation detail rather than a design
   question.

## 12. What This Design Explicitly Rejects

Listed so future maintainers (probably future-Todd) don't relitigate them.
Each rejection has a one-line reason; longer reasoning is in the
conversation history that produced this document.

- **In-process embedding model** (sentence-transformers loaded into the
  search process). Rejected: Ollama is committed-to as permanent
  infrastructure, so in-process models are duplication of a service we
  already run.

- **Direct sqlite-vec without txtai.** Rejected: txtai-on-sqlite-vec gets
  us both at once, plus SQL filters, hybrid search, RAG, and a workflow
  framework as configuration rather than code.

- **Per-keystroke fzf reload.** Rejected: it was an experiment, not a
  requirement. Search-on-enter is a perfectly fine UX and dissolves the
  cold-start latency concern that originally motivated daemon-vs-no-daemon
  arguments.

- **Spans in the index.** Deferred per §7. Re-introducible without a schema
  change if empirical retrieval quality demands it.

- **Schema migrations.** Rejected: SOT is on disk, reindex is cheap, schema
  bumps are `DROP TABLE; reindex`.

- **Ollama failover, retry, queueing, backoff, circuit breaker.** Rejected
  per §3 rule 6 and §9.4. Fail loud. The filesystem is the queue.

- **Per-pane DB-tracked ingestion state.** Rejected in favor of `.indexed`
  marker files. Filesystem markers are atomic, inspectable with `ls`,
  trivially resettable, and survive index destruction.

- **Custom IPC protocol** (UNIX socket, FIFO, newline-delimited JSON over
  stdin/stdout). Rejected: the daemon already speaks HTTP for indexing,
  there's no reason to invent a second transport for search.

- **The hexagonal port as a swap-readiness mechanism.** Rejected as
  framing. The port is retained as *insulation* between the domain and
  txtai's API surface, not as preparation for replacing txtai. We are not
  replacing txtai.
