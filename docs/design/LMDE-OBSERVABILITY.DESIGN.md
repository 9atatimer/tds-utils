## Overview

The LMDE Observability Stack provides a local, permanent, and secure destination for metrics and traces generated by coding agents (e.g., `clai`) and other local services. It utilizes a `kind` cluster for isolation, a local container registry for supply-chain resistance, and industry-standard tools (OTel, Prometheus, Grafana) for the data pipeline. The stack's UIs are reached from the host at stable `*.lmde.localhost` vhosts through Caddy and an in-cluster `ingress-nginx` controller.

---

## Goals

1. **Persistent Metrics** -- Metrics survive `kind` restarts, host reboots, and cluster recreations via host-path mapping.
2. **Supply-Chain Resistance** -- All images are pinned by SHA256 digest and served from a local registry (`localhost:5001`).
3. **Unified Telemetry Endpoint** -- A single OTel collector endpoint (`localhost:4317`/`4318`) for all local telemetry.
4. **Blast Radius Control** -- Resource quotas and limits ensure observability services do not exhaust host resources.
5. **Zero-Configuration for Consumers** -- Any tool on the laptop can assume the OTLP endpoint is available.
6. **Named Host Access** -- Grafana, and any future LMDE cluster service, is reachable at a stable `<service>.lmde.localhost` URL that survives pod restarts, rollouts, and cluster recreation.

---

## Non-Goals

- **Log Aggregation** -- This stack focuses on metrics and traces. Log management is handled by `log-hoarder`.
- **Multi-node Scalability** -- Designed for single-user laptop scale.
- **Public Ingress** -- The stack is strictly local and not exposed to the internet.
- **Service Mesh** -- No mTLS, traffic-shaping, or east-west policy between observability components. Routing is north-south only (host -> cluster).

## Design

### 1. Local Container Registry

Ensures that the environment is hermetic and resistant to upstream poisoning or outages.

#### Responsibilities

| Responsibility | Details |
|----------------|---------|
| Image Mirroring | Pulls upstream images, verifies digests, and pushes to `localhost:5001`. Includes the `ingress-nginx` controller and admission-webhook images. |
| Digest Pinning | All images are referenced by `name@sha256:hash` in `images.txt`. |
| Lifecycle | Started as a standalone Docker container before `kind` to avoid circularity. |

### 2. Infrastructure (kind)

Provides the isolation boundary and resource management.

#### Configuration (`kind-config.yaml`)

- **Nodes**: 1 control-plane node, labeled `ingress-ready=true` so the `ingress-nginx` controller schedules onto it (required by the kind ingress pattern).
- **Port Mappings**:
  - `80` (node) <-> `127.0.0.1:32100` (host) -- the cluster's single HTTP host-edge door, served by `ingress-nginx`.
  - `4317` <-> `4317` (OTLP gRPC)
  - `4318` <-> `4318` (OTLP HTTP)
  - The previous direct `3000` Grafana mapping is **removed** -- Grafana is reached through `ingress-nginx`, not a dedicated host port.
- **Mounts**:
  - `~/.local/share/tds-utils/observability/data` -> `/mnt/data` (Persistence)

### 3. Telemetry Pipeline (OTel + Prometheus + Grafana)

The core data processing and visualization stack.

#### Deployment Strategy

- **Helm**: Use `prometheus-community` and `grafana` charts, but template them to use local registry images.
- **Persistence**: Configure charts to use a `HostPath` PersistentVolume pointing to `/mnt/data`.
- **Quotas**: A `ResourceQuota` in the `observability` namespace limits CPU to 1.0 and Memory to 2Gi.
- **Grafana Service**: `ClusterIP` only -- reached exclusively through `ingress-nginx`, never via a host port. Dashboards are provisioned from `specs/grafana/dashboards/`.

### 4. Host Ingress (Caddy + ingress-nginx)

Bridges the host to in-cluster HTTP services so they answer on stable, named vhosts. The pattern is reusable across LMDE clusters:

```
*.{cluster}.localhost  ->  Caddy  ->  ingress-nginx (in {cluster})  ->  Service  ->  pod
```

#### Responsibilities

| Responsibility | Details |
|----------------|---------|
| TLS + host edge | Caddy owns host `:80`/`:443`, terminates TLS for `*.lmde.localhost` (internal CA), and reverse-proxies to the cluster's loopback port. |
| Cluster routing | `ingress-nginx` routes by `Host` header to the target `Service`; pod churn is absorbed via the Service's EndpointSlices with zero reconfig and no reload. |
| Host boundary | One kind `extraPortMapping` publishes `ingress-nginx`'s node port to a per-cluster host loopback port (`127.0.0.1` only). |
| Vhost registration | `networking/lib.sh` registers/updates the Caddy route for `*.{cluster}.localhost` idempotently via the Caddy admin API. |

#### Why the pattern scales

- **One Caddy route per cluster** -- `*.lmde.localhost -> 127.0.0.1:32100`. Not per service.
- **One `extraPortMapping` per cluster** -- frozen at cluster-create time, so it must be allocated once and never needs to change.
- **A new vhost in an existing cluster is one `Ingress` object** -- no host-side change, no cluster recreate.
- **A new cluster** -- allocate the next free host loopback port, add one Caddy route, install `ingress-nginx`.

Because each `kind` cluster is its own set of Docker containers, two clusters cannot publish to the same host port; each cluster gets a unique host loopback port. Host ingress ports follow the `3210X` convention -- `X` is the cluster index `0`-`9` -- drawn from the Kubernetes NodePort range (`30000`-`32767`) so they clear the common dev ports and stay below the Linux ephemeral floor (`32768`). The observability cluster is index `0`, hence `32100`. The `lmde/components/networking/` component owns port allocation and the Caddy registration helper.

## State Machine: Bootstrap Flow

```
Registry --> Kind Cluster --> Ingress Controller --> Telemetry Stack --> Host Routing --> Readiness
```

| From | To | Trigger | Condition |
|------|-----|---------|-----------|
| START | Registry | `setup.sh` | Docker is running |
| Registry | Kind Cluster | Registry is healthy | `images.txt` is synced |
| Kind Cluster | Ingress Controller | Cluster is Ready | `ingress-ready` node label present |
| Ingress Controller | Telemetry Stack | `ingress-nginx` admission webhook Ready | HostPath dir exists |
| Telemetry Stack | Host Routing | Grafana + Prometheus pods Running | Grafana Service has endpoints |
| Host Routing | Readiness | `Ingress` applied + Caddy vhost registered | `curl grafana.lmde.localhost` returns 200 |

## Security Considerations

- **Supply Chain** -- Mitigated by local registry + digest pinning.
- **Resource Exhaustion** -- Mitigated by K8s ResourceQuotas.
- **Data Privacy** -- All data stays on the local disk; no "phone home" analytics in Grafana/Prometheus.
- **Ingress Exposure** -- `ingress-nginx` is published only to host loopback (`127.0.0.1`) via the kind `extraPortMapping` `listenAddress`; it is never bound to a routable interface.

## Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Storage | HostPath Mapping | Simplest path to survival across cluster deletes in `kind`. |
| In-cluster HTTP routing | `ingress-nginx` | A controller, not a config file: routes by `Host` to Services, absorbs pod churn with zero reconfig, and one mechanism scales to many vhosts. |
| Host-to-cluster path | One kind `extraPortMapping` per cluster -> `ingress-nginx` hostPort | Each `kind` cluster is isolated Docker containers; a unique host loopback port per cluster is the only stable door. |
| Ingress host port | `3210X` (X = cluster index `0`-`9`); observability = `32100` | Within the Kubernetes NodePort range `30000`-`32767`: clear of common dev ports, below the Linux ephemeral floor (`32768`). |
| Vhost scheme | `*.{cluster}.localhost` (this cluster aliased `lmde`) | `.localhost` resolves to loopback with no dnsmasq config; a per-cluster wildcard keeps it to one Caddy route per cluster. |
| Registry | `Registry:2` (standalone) | Avoids circular dependency where the cluster needs the registry to start the registry. |
| Deployment | Helm (Local Images) | Leverages community best-practices for config while maintaining image control. |

## Rejections

- **Istio (service mesh)** -- Its ingress gateway is itself an in-cluster Service, so it inherits the same host-boundary problem without removing it, while adding istiod, sidecars, and CRDs; mesh features are a Non-Goal.
- **Direct NodePort per service (no ingress controller)** -- Works for one service, but every new vhost needs a new NodePort *and* a new `extraPortMapping`, and `extraPortMappings` are frozen at cluster-create time -- so each new service would force a cluster recreate.
- **In-cluster ingress as the host edge (skip Caddy)** -- Caddy is the LMDE platform's single host reverse proxy/TLS terminator; the cluster sits behind it on a loopback port rather than competing for host `:80`/`:443`.
- **Dynamic Provisioning (EBS/GCP)** -- Not applicable to a local laptop environment.
- **Local Registry as a Pod** -- Circular dependency risk.

## Future Considerations

- **Alerting** -- Integrating with macOS notifications via a custom exporter.
- **Tracing** -- Adding Jaeger or Tempo if traces become as important as metrics.
- **Generic cluster-ingress mechanism** -- The `*.{cluster}.localhost` pattern is implemented in `lmde/components/networking/` for the observability cluster. When a second LMDE cluster appears, promote it to a dedicated `LMDE-NETWORKING.DESIGN.md` with a reusable `register_cluster_vhost` helper and a host-port allocation registry.
- **Cluster rename** -- The vhost alias is `lmde` while the `kind` cluster is `lmde-observability`. Renaming the cluster to `lmde` would align them but is cosmetic and deferred.
